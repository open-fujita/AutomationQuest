# das-spec-notes — 青ロボット（Basic Engine Robot）と緑ロボット（Robot / DA）の STEP 操作・フロー制御 仕様ノート

> 親 INDEX: [[INDEX]]

第2弾「緑ロボット編」実装のための一次情報パッケージ。公式 docshield（ja）と Qiita ハンズオン記事を 2026-06-12 に調査した結果を整理する。実装エージェントはこのノートを典拠として UI 文言・意味論を実装すること。

## 1. 2種類のロボットの違い（公式「ロボット構築の概要」より）

| 観点 | 青ロボット（Basic Engine Robot / ベーシックエンジンロボット） | 緑ロボット（Robot / Desktop Automation） |
|---|---|---|
| 主対象 | 静的 HTML ページの自動化 | 動的 Web サイト、Windows / Java アプリ、ターミナル、Citrix |
| 状態の所在 | **ロボットが内部追跡**（ロボット状態がステップ間を流れる） | **アプリケーション側（外部）**にある |
| 実行方向 | 分岐・状態巻き戻しあり（バックトラック型） | **前方への移動のみ**（戻りなし） |
| 分岐 | 分岐ポイントで複数分岐を順次実行 | トップレベル分岐なし。**複合ステップ内のみ**（条件 / ガードチョイス / Try-Catch） |
| エディタ | 左→右フローチャート（ロボットビュー＋ブラウザビュー） | **縦のワークフローツリー**＋レコーダービュー |
| 連携 | 「**ロボットを呼び出す**」アクションステップで緑ロボットを呼び出して再利用できる | 呼び出される側（入力/出力変数で契約） |

## 2. 青ロボット（Basic Engine Robot）の STEP・フロー制御

第1弾で実機 .robot XML から確立した意味論（[[ds-real-ui-notes]]）と公式記述が一致することを確認済み。

### 2.1 実行モデル（接続と実行フロー）
- 「ロボット状態」が生成され、最初のステップへの入力になる。各ステップ = **入力状態 → ファインダー適用 → ステップアクション実行 → 出力状態を生成**。
- **ループ**: ループ系アクション（要素の繰り返し 等）は入力状態から**複数の出力状態を生成**し、後続ステップが反復実行される。実行順は `A, B[1], C, D, B[2], C, D, B[3], C, D`。
- **分岐ポイント**: 1 ステップから複数ステップへ接続できる。「分岐ポイントから出るすべての分岐は次々に実行」され、番号指定がなければ**上から順（トップダウン）**。各分岐の開始時にロボット状態は分岐時点に戻る。
- ステップは実行時に出力状態を生成しないことがある（エラー / テスト不成立で実行が別ステップへ移る場合）。

### 2.2 ステップの種類と構造
- ステップ4種: **アクション / トライ / グループ / 終了**（終了＝分岐の終了マーク。End 到達で「呼び出し元」＝直近の分岐ポイントの次分岐 or ループの次反復へ復帰）。
- アクションステップの構造 = **ステップ名＋ファインダーリスト（タグファインダー/範囲ファインダー）＋ステップアクション＋エラー処理**。プロパティペインの 4 タブ（基本/ファインダー/アクション/エラー処理）に対応。

### 2.3 トライステップとエラー処理（条件とエラー処理）
- トライステップは分岐ポイントに似るが、**先頭分岐のみ実行**。分岐内のステップがエラーを出し、そのステップのエラー処理が「**次の代替を試行**」なら、**トライステップから出る次の分岐の先頭**から実行を続行（`A, T, B →エラー→ T, D, E`）。
- **テストステップの条件不成立も「エラー」と同一機構**で処理される（特定のエラーはある状況では失敗した条件、別の状況では実エラー — 境界を区別せず統一処理するのが設計思想）。
- プログラミング言語との対応: 先頭+最終分岐 = if-then-else / 複数分岐 = else-if / エラー検出 = try-catch。
- 定石: トライ3分岐（構造A用 → 構造B用 → **必ず成功するデフォルト**）。
- エラー処理プロパティの選択肢:
  - 報告: エラーを報告して停止（既定）/ **API 例外として報告**（RobotErrorResponse として呼び出し元へ）/ エラーとしてログ記録
  - Then（続行方法）: **次の代替を試行** / **後続のステップをすべてスキップ** / **ループの次の反復** / 停止

## 3. 緑ロボット（Robot / Desktop Automation）の STEP・フロー制御

### 3.1 アーキテクチャ（DAS）
- **Desktop Automation サービス（DAS）**: 自動化対象（オートメーションデバイス）の Windows マシンにインストールするサービス。Windows のみサポート。Design Studio / Management Console / RoboServer からデバイスマッピングで接続する。
- 対象: ネイティブ Windows アプリ / Java アプリ / レガシーターミナル / Citrix。組み込み Chromium での Web、組み込み Excel ドライバーもある。
- 制限: 整合性レベル「高」のアプリ（タスクマネージャ等）にはツリー生成・入力生成不可（DAS の管理者実行か仮想入力ドライバーで回避）。
- **ツリーモード**（アプリ要素ツリーの生成方法）: **ISA（インテリジェント スクリーン オートメーション）**＝画面認識ベース（API 非対応アプリ/Citrix/RDP/SAP 向け、der_x 等の座標属性）/ **Windows オートメーション API**（既定。Excel/IE/Java/SAP 拡張サポート）/ **ツリーなし**。アプリ全体・コンポーネント単位で切替可。「非表示要素のスキップ」でツリー抽出を高速化。

### 3.2 エディタ画面
- **ロボットビュー（ワークフロー）**: ステップを縦に並べたツリー。**現在のステップは緑色**でマーク。ステップをクリックすると**そこまでロボットが実行**され、そのステップが新しい現在ステップになる。「現在の実行パス」が可視化される。複数選択は Ctrl+クリック。
- **レコーダービュー**: アプリ画面のライブ表示＋**利用可能なエレメントのツリー**をタブ表示。**要素を右クリック → ステップ挿入**（クリック/値を抽出/ループ 等。挿入時にファインダーが自動生成される）。下部にマウス座標（ウィンドウ左上基準）とデバイス状態のライブストリーミング表示。

### 3.3 ファインダー（4階層）
1. **デバイスファインダー**: デバイスセレクターのみ（最も単純）。
2. **アプリケーションファインダー**: デバイス → アプリを選択。アプリケーションセレクターは CSS 構文。検出済みアプリの内部ハンドルで複数インスタンス対応。
3. **コンポーネントファインダー**（最も一般的）: 入力フィールド/ボタン/テーブル等を検索。CSS 構文のコンポーネントセレクター＋「内側のコンポーネント」＋テキストマッチ（Regex）。
4. **イメージファインダー**: 見た目で検索（レコーダービューで矩形ドラッグ）。RDP の「接続品質の自動検出」はピクセル一致を壊すので品質固定を推奨。9 グリッドイメージファインダーは廃止予定。

- **セレクタ構文**（CSS 風）: `elementName[attributeName="value"]`。演算子: `^=`（前方一致）/ `$=`（後方一致）/ `*=`（部分一致）/ `:nth-child(n)` / `E > F`（子）/ `E + F`（直後）/ `E ~ F`（後続）。複数属性の AND: `button[visible="true"][name^="Save"]`。
- **再利用**: コピー＆ペースト（最も脆い）/ **前のファインダーを参照**（「(直前の)」表示、最も一般的）/ **名前付きファインダー**（エイリアス）。

### 3.4 ガードチョイス（安定化の要）
- 複数の**ガード（条件）と関連アクション**を設定し、**並行監視して最初に成立したガードの枝のみを排他実行**する（他のガードは評価対象外になる）。
- ガード7種:

| ガード | 動作 |
|---|---|
| **時間経過（Timeout）** | 指定秒数の経過で成立（**既定 60 秒**）。フォールバック用 |
| **該当するロケーション（Location Found）** | ファインダーがエレメントを検出したら成立 |
| **該当しないロケーション（Location Not Found）** | エレメントが見つからない場合に成立 |
| **取り除かれたロケーション（Location Removed）** | エレメント検出後、それが除去されるまで待機 |
| **該当するアプリケーション（Application Found）** | アプリケーションが見つかるまで待機して成立 |
| **該当しないアプリケーション（Application Not Found)** | アプリ未検出で成立 |
| **ツリーの変更停止（Tree Stopped Changing）** | ツリー変更後、指定**ミリ秒**間の静止を確認して成立 |

- ガードはガードフレーム下部の**緑のプラス記号**で追加。Location/Application 系はファインダー必須。
- 推奨パターン: 本命ガード（例: フィールド値検証）＋ Timeout ガードの 2 本構成で、不完全な状態での後続実行を防止。
- **教材上の核心**: 「固定秒待ち（時間経過のみ）」は遅い日に壊れる → 「状態を待つ（Location Found）」へ転換させる。

### 3.5 ループ
- **Loop ステップ**: ステップのグループを繰り返す。**Break**（ループ終了）/ **Continue**（次の反復へスキップ）と組合せ。条件ステップやガードチョイスをループ内に置き「状態変化まで待ちながら回す」が定石。
- **For Each ループ**: アプリツリーのノード群を反復。
  - **スコープファインダー**: 反復の起点となるツリー部分を特定（ステップ内で一意の名前必須）。
  - **エレメントファインダー**: スコープファインダー名＋**相対セレクタ**（例 `> DIV`）で反復対象を検出。合成セレクタは `DIV[class='someClass'] > DIV` のようになる。
  - 反復中にツリーが変化して新要素が出た場合、現在位置との相対位置次第で反復に含まれたり含まれなかったりする。
  - **ループ内のファインダーは検出要素からの相対参照**になり、反復ごとに堅牢。
  - レコーダービュー右クリック「ループ」サブメニュー: **すべての兄弟** / **各[タグ名]兄弟** / 各[タグ名](レベル)親（組み込みブラウザのみ）/ **テーブル行の反復**（組み込みブラウザのみ）。
  - スキップは条件ステップ＋式（例 `=i % 2 == 0`）。

### 3.6 条件・Try-Catch・Throw
- **条件（Condition）**: ブール条件＋分岐のリスト。**最初に真と評価された条件の分岐のみ実行**、他はスキップ。該当なしなら何もせず次ステップへ。`true` 条件で常時実行のデフォルト分岐（if-then-else 相当）。
- **Try-Catch**: **Try / Catch（複数可・例外種別ごと）/ Finally** の3部構成。
  - 定義済み例外: `TimeOutError` / `FinderIssue` / `DeviceIssue` / `IncorrectValueIssue` / `ExtractIssue` / `DivisionByZeroIssue` / `OverFlowIssue` / `ConversionIssue` / `DisappearedIssue`。
  - 例外発生 → 該当する Catch 分岐の先頭へ直接移行。該当 Catch がなければ外側の Try-Catch へ伝播、なければロボット終了。**一度キャッチした例外は外側で再処理されない**。
  - Finally は Try/Catch の結果に関わらず実行（クリーンアップ用）。Finally 内の例外は未処理例外を上書きする。
- **Throw**: 明示的に例外を送出（カスタム例外可）。

### 3.7 主要ステップカタログ（公式 c_dassteps.html 全分類）
- **代入/変換**: Assign / Convert Value（式 expression）
- **制御フロー**: 条件 / Try-Catch / Throw / **ガードチョイス** / グループ / Return / Trigger Choice（イベント駆動分岐）
- **ループ**: Loop / While Loop / **For Each** / DB クエリループ / メールストリーム / ディレクトリ反復 / Break / Continue
- **アプリ統合**: ブラウザ（組み込み Chromium）/ ウィンドウ（Windows アプリ起動）/ Excel（組み込みドライバー）/ ターミナル / Document Transformation / PDF / メール
- **入力操作**: **クリック**（左/中/右、回数=ダブルクリック2、修飾子 Shift/Ctrl/Alt、オフセット=9基準点＋px）/ **テキストを入力** / キープレス / マウスプレス（ドラッグ）/ マウス移動 / スクロール
- **抽出**: **値を抽出**（属性 / 拡張属性（der_ 接頭辞なし指定）/ テキスト（下位層含むオプション）→ 式で変換 `$initial.integer()` 等 → 変数へ格納。複数保存で姓名分割等も可）/ ツリーを XML 抽出 / イメージ抽出 / イメージからテキスト抽出（OCR）/ クリップボード
- **DB/ファイル**: クエリ/格納/検索/削除/SQL 実行 / ファイル読み書き / ファイルシステム操作
- **デバイス管理**: デバイスへ接続 / 切断 / リモートデバイスアクション / RDP ログイン
- **出力**: Output Value / ログ書込 / 通知
- **その他**: Freeze Tree（ツリー固定）/ カスタムアクション / KTA / Cloud AI
- **「開く」ステップは非推奨**（ブラウザ/Excel/ウィンドウ/ターミナル等の専用ステップに分割済み）。URI 形式の名残: `excel://new`、`rdp://user:pw@host` 等。

## 5.5 【2026.1 忠実化追補】実機 UI とステップカタログ（2026-06-12 藤田さん指摘により追加）

藤田さんの実機スクリーンショット（DS 緑ロボット編集画面）と 2026.1 公式ドキュメント
（`https://docshield.tungstenautomation.com/RPA/ja_JA/2026.1-cqvh2o7vk9/help/rpa_help/help_main/designstudio/`）で確認した **正** の仕様。
初版実装（縦ツリー＋右プロパティペイン）は誤りであり、以下に置き換える。

### A. ワークフロー描画（実機準拠・最重要）
- **横方向フロー**: `○（フローポイント）—[ステップカード]—○—[ステップカード]—○` を左→右に連結。青い接続線。
- **ステップカードは折りたたみ⇔展開**: 折りたたみ時=アイコン＋名前＋▼（アクション切替ドロップダウン）。展開時=**カード内にそのステップの設定フォームがインライン表示**（ヘッダ: アイコン＋タイトル＋ ^（折りたたみ）＋ ?（ヘルプ））。右側プロパティペインでの編集ではない。
- 設定不備のステップは**黄色の ⚠ バッジ**をカード右上に表示。
- 選択中ステップは**緑の枠線**。
- **ガードチョイス**（公式画像 GuardedChoiceLocation.png / add_guard2.png より）:
  - カードヘッダ: 🖐（手のひら）アイコン＋「ガード チョイス」＋^＋?
  - カード内に**ガードレーンが縦に並ぶ**。各レーン = ガード種別ドロップダウン（表示例「ロケーション...」「秒が経過した...」）＋ガード設定のインラインフォーム → 青線 → ○ → そのガード枝の後続ステップカード（例: ❗「スロー TimeOutError」、例外=TimeOutError）→ ○
  - レーンの間に**破線＋緑の ⊕** が現れ、クリックでガード追加。

## 6. ゲーム実装への含意（要点）

1. **緑ロボのエディタは新規 UI**（縦ワークフローツリー＋レコーダービュー）。第1弾の React Flow フローチャートは使わない。
2. **シミュレータは tick 駆動**にし、「待ち」を一級概念にする。ガードチョイスの意味論（並行監視・最初成立・排他実行・Timeout 既定60秒）を忠実に実装。
3. **ファインダーは CSS 風セレクタの実物表記**を見せる。D5 で「座標固定は壊れる／属性セレクタは生き残る」を体験。
4. **For Each はスコープ＋相対セレクタ**の2段構造を UI でそのまま見せる（実機の右クリック「すべての兄弟」等も再現）。
5. 用語集追加候補: 緑ロボット / DAS / オートメーションデバイス / レコーダービュー / ファインダー / ガード / ガードチョイス / ツリーの変更停止 / スコープファインダー / ツリーモード / Break・Continue / 例外（FinderIssue 等）。
