# requirements.md

> 親 INDEX: [[INDEX]]

## 概要

「自動化推進室クエスト」第1弾（青ロボット＝Basic Engine Robot 編、相談 M1〜M5）に続き、第2弾として緑ロボット（Robot / Desktop Automation）編の相談 D1〜D5 を追加する。新たなロボット種別・エディタ UI・シミュレータ・ミッション定義を既存コードを壊さず後方互換に実装する。

---

## 背景・動機

第1弾は Web スクレイピング文脈の「青ロボット（Basic Engine Robot）」を学習対象とし、クリア済み。現場で頻出するもう一方の主役「緑ロボット（Robot / Desktop Automation、DAS）」は、Windows / Java アプリなどのデスクトップ自動化を担い、ガードチョイス・For Each ループ・CSS 風セレクタなど青ロボとは全く異なる操作モデルを持つ。Qiita ハンズオン記事（@ParakeetOnTheHead）に示された実務課題（ガード同期・動的リスト・レイアウト変動）を題材に、「状態を待つ」「スコープを絞る」「セレクタで追う」という緑ロボの三大思考を体験できるゲームを届けたい。

---

## ユーザーストーリー

- As a BizRobo! 学習者、I want 青ロボとは別のエディタ UI で緑ロボのワークフローを組む体験をしたい、so that デスクトップアプリ自動化の基本操作（開く・記録・クリック・抽出）を直感的に習得できる。
- As a BizRobo! 学習者、I want ガードチョイスによる「状態待ち」の仕組みを模擬アプリで体験したい、so that 「固定秒待ち」が遅い日に壊れる理由と Location Found による同期の優位性を理解できる。
- As a BizRobo! 学習者、I want ランダムに出現する通知ウィンドウを Application Found ガードで捌くシナリオを試したい、so that 不測の割り込みを柔軟に処理する実務感覚を養える。
- As a BizRobo! 学習者、I want 動的に件数が変わるリストを For Each ループで全件取得する体験をしたい、so that スコープファインダーと相対セレクタの役割を身に付けられる。
- As a BizRobo! 学習者、I want レイアウト変動するアプリで CSS 風セレクタと座標固定の耐久性を比べたい、so that 属性セレクタが壊れにくい理由と設計判断の根拠を習得できる。

## 受け入れ条件

### 全 D 相談共通

- Given ホーム画面が表示されている、When 「緑ロボット編」の相談を選択する、Then 相談票（D1〜D5 の依頼文）が表示され、見立てクイズ→ワークフロー組み立て→実行→判定の標準フローでプレイできる。
- Given 第1弾相談（M1〜M5）、When 任意の M 相談をプレイする、Then 挙動・見立て・判定結果が第2弾追加前と変わらない（リグレッションなし）。
- Given いずれかの D 相談をクリアする、When クリア画面が表示される、Then クリア時に解禁される用語集キーが glossary に登録され、プロフィール別の localStorage 進捗に記録される。
- Given プロジェクト全体、When `npm run typecheck && npm run build && npm run test` を実行する、Then 型エラー・ビルドエラー・テスト失敗が 0 件。

### D1「はじめての緑ロボット」

- Given シミュレータを起動する、When 「開く」相当ステップ（ウィンドウを起動）・レコーダービューの模擬表示・クリックステップ・値を抽出ステップを含むワークフローを組む、Then 模擬デスクトップアプリから目標値が正しく抽出されシミュレーション成功となる。
- Given プレイヤーが受け入れ条件を充足した構成を組む、When 実行する、Then 構造ベース判定（ステップ種別・アクション種別の存在確認）でパスし、ステップ名の完全一致は要求されない。

### D2「待ち方を覚える」

- Given 模擬アプリがボタン有効化に遅延（tick 数可変）を持つ、When 固定秒待ち（時間経過のみのガード）で構成する、Then 遅延が長い実行シードで失敗する（固定秒の脆さを体験できる）。
- Given 同じ模擬アプリ、When Location Found ガード＋ Timeout ガードの 2 本構成（ガードチョイス）を組む、Then 遅延長短どちらのシードでも成功し、Timeout はフォールバックとして機能する。
- Given ガードチョイスのシミュレーション、When 複数ガードが並行監視される、Then 最初に成立したガードの枝のみ排他実行され、Timeout 既定 60 秒が正しくモデル化される。

### D3「不意の来客」

- Given 模擬アプリが tick ランダム（シード決定的）で通知ウィンドウを出現させる、When Application Found ガードを含まない構成で実行する、Then 通知ウィンドウが出た試行で失敗する。
- Given 同じ模擬アプリ、When Application Found ガードで通知ウィンドウを検知・閉じる枝を設ける、Then 通知有無どちらのシードでも成功する。

### D4「動くリストを数える」

- Given 模擬アプリのリスト件数が tick ごとに変動する（シード決定的）、When For Each ループをスコープファインダー＋相対セレクタで設定する、Then 全件を正しくループし抽出件数がリスト実件数と一致して成功する。
- Given For Each なしまたはスコープ未設定の構成、When 実行する、Then 1 件のみ取得または件数ミスマッチで失敗し、ヒントが表示される。

### D5「要素を見失わない」

- Given 模擬アプリの列順がシード決定的にシャッフルされる、When 座標固定（絶対座標）で抽出ステップを設定する、Then 列順変動時に誤った値を取得して失敗する（壊れる体験）。
- Given 同じ模擬アプリ、When CSS 風セレクタ（属性ベースのコンポーネントセレクタ）で抽出ステップを設定する、Then 列順変動後も正しい値を取得して成功する。

## スコープ

### In Scope

- 緑ロボット専用エディタ UI（縦ワークフローツリー＋レコーダービューの模擬）の新規実装
- 新規ステップアクション型（DAS 系: Open / Click / ExtractValue / GuardChoice / ForEach / Condition / AppGuard / Selector 等）の model/robot.ts への後方互換追加
- tick 駆動シミュレータの拡張（DAS 実行モデル: 前方移動のみ・ガードチョイス並行監視・For Each スコープ）
- D1〜D5 ミッション定義ファイル（src/data/missions/d1.ts〜d5.ts）と index への登録
- 模擬デスクトップアプリ描画（タイトルバー付き架空 Windows アプリをブラウザ内 UI で実現）
- D1〜D5 の見立てクイズ（DeductionQuestion）・受け入れ条件（MissionCheck）・クリア時 reveal 実装
- 用語集（glossary.ts）への緑ロボ関連用語追加（das-spec-notes §6 候補リスト準拠）
- D シリーズ向けバリデータ拡張（構造ベース: ガード種別・セレクタ型・For Each スコープ有無の確認）
- M1〜M5 のリグレッションテスト維持（Vitest）
- D1〜D5 の新規ユニットテスト（シミュレータ決定性・バリデータ正否）

### Out of Scope（重要）

- 新規 npm パッケージの追加（依存は既存のまま固定）
- 実機 Design Studio / DAS サービスへの接続・通信
- 第3弾相談（Gd1「Tree Stopped Changing」等）の実装（第3弾候補として記録のみ）
- モバイル向けレスポンシブ最適化（既存方針踏襲）
- Backlog チケットや CI 設定の変更（統括が担当）
- git commit / push（統括が実施）
- docs/ 永続ドキュメント（product-requirements.md 等）の新規作成（現プロジェクトは docs/ 未整備。整備は別タスク）

## 制約事項

- **技術的制約**: React 18 / TypeScript / Vite 6 / @xyflow/react / Tailwind / Zustand / Vitest のみ使用。新規依存禁止。`npm install -g` 禁止。
- **実行環境**: Node 18 以上（Node 16 以下では Vite 6 が起動しない）。
- **決定的シミュレーション**: シード乱数を使い Date.now / Math.random を直接呼ばない。模擬アプリの状態遷移（ボタン有効化遅延・通知出現・リスト件数変動・列順シャッフル）は tick タイムラインで完全決定的であること。
- **正規用語の使用**: UI 文言・ヒント・クイズ・用語集はすべて公式日本語用語に従う（ガードチョイス / 該当するロケーション / ツリーの変更停止 / レコーダービュー / ファインダー 等。das-spec-notes §3.4 準拠）。ガード種別の名称は公式表記を使い独自造語しない。
- **後方互換**: Mission 型・Robot 型への追加フィールドはすべて optional。既存の M1〜M5 データが型エラーなくコンパイルされること。
- **構造ベース判定**: 受け入れ条件はステップ名の文字列一致を要求せず、アクション種別・ガード種別・セレクタ型の構造で判定する（第1弾 validator.ts の設計思想踏襲）。
